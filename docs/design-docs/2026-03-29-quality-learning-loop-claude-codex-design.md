## Problem Statement

Belayer orchestrates autonomous coding agents through YAML pipelines, but has no memory across runs. When an adversarial review gate fails, the failure output is used for retry but never persisted. The same mistakes recur across sessions, branches, and features. A developer working with belayer today gets the same quality of implementation prompts on day 90 as day 1.

The v5 direction separates three roles (Agent, Harness, Orchestrator) and identifies belayer as the orchestration layer only. Within that framing, the highest-value unsolved problem is the quality learning loop: capture gate failure patterns, store them as quality knowledge, and inject relevant patterns into future implementation prompts. Every failed gate makes future runs better.

## What Makes This Cool

A flywheel that no single-agent tool can have. Claude Code, Codex, Slate... they're all ephemeral. Fresh context every session. Belayer sits above them, persists quality patterns across sessions, and makes every agent it orchestrates get smarter about THIS repo over time. The agent doesn't need to know belayer's learning system exists. From the agent's perspective, it just gets a slightly better prompt each time. The flywheel is invisible to the agent but visible in the gate pass rate.

Codex's independent review called this a "scar-tissue engine" and proposed that learnings don't just inform prompts, they become dormant micro-gates that wake up when future diffs hit matching paths. Codebase-specific immune system.

## Constraints

- Go CLI. New code goes in `internal/quality/` and `internal/cli/quality.go`
- Must work with any framework (claude-tmux, claude-codex, custom). Quality system is framework-agnostic.
- Learning extraction from gate failures needs a cheap model call (Haiku-class) to structure the raw failure output. One call per gate failure.
- Storage must be concurrent-branch-safe (multiple pipelines running simultaneously)
- No external dependencies (no vector store, no embedding model, no database beyond filesystem)
- Must integrate with existing gate system (`gate-result.json` + `rationale.md`)
- The claude-codex framework is the second deliverable, built on top of the quality system

## Premises

1. Belayer is orchestration-only. The existing codebase has the right bones (ExecSpawner, pipeline YAML, gates). This requires a new package + CLI commands, not a rewrite.
2. `belayer quality` is the highest-value new Go code. The unique moat. Build this before the framework.
3. H-as-feature means no episode compression work. Node handoffs are intentionally thin.
4. Existing intake (`belayer start` + `explorer:send`) is good enough for MVP.
5. Quality patterns store "what makes work fail review in this repo" not "how the codebase works."

## Cross-Model Perspective

**Coolest version:** "Scar-tissue engine. Every failed gate becomes a dormant micro-gate with path triggers, a failure signature, one hard adversarial question, and optionally an executable detector. When a future diff hits matching paths, inject the lesson AND wake up that micro-gate as a targeted reviewer. Codebase-specific immune system."

**Key insight:** "The exciting part is not YAML, Temporal, or agent orchestration. It is compounding judgment: every miss becomes sharper future taste."

**50% existing tool:** Semgrep Community Edition. YAML-native rules, path scoping, severity, deterministic enforcement. Its rule syntax maps cleanly to "quality patterns as YAML." Missing 50%: mining gate failures into patterns, ranking by path relevance, injecting into prompts, handling recurrence/staleness/drift.

**Weekend build priority:** "Thinnest closed loop. Emit immutable failure signals. Add `belayer quality check` returning top 3 recurring failures by path overlap. Generate `quality-context.md` prepended to implementation prompt. Add `belayer quality stats` with recurrence count only. Do NOT build age decay, drift detection, or compaction first."

## Approaches Considered

### Approach A: Framework-First
Ship `frameworks/claude-codex/` with Claude implementing and Codex reviewing adversarially. Zero new Go code. Effort: S. Risk: Low. Proves end-to-end but no learning loop (no moat).

### Approach B: Quality Loop First (CHOSEN)
Build `belayer quality` CLI commands, learning storage in `internal/quality/`, signal system for concurrent safety. Then create claude-codex framework that uses it. Effort: M. Risk: Medium (designing format before real data). Ships the moat from day one.

### Approach C: Framework + Shell Script Quality
Ship both, but quality system is shell scripts in the framework. Graduate to Go later. Effort: S-M. Risk: Low-Medium. Functional but hacky, will need rewrite.

## Recommended Approach

**Approach B: Quality Loop First.** The quality loop IS the product. Build it right in Go, then the framework demonstrates the full flywheel on first use.

Informed by Codex's weekend build advice: start with the thinnest closed loop. Emit signals, retrieve by path overlap, inject into prompts. Defer age decay, drift detection, compaction to post-MVP.

### Phase 1: Quality System (internal/quality/ + CLI)

**New package: `internal/quality/`**

`learning.go` — Core data model:
```go
type Learning struct {
    ID          string   `yaml:"id"`
    Created     string   `yaml:"created"`
    SourceGate  string   `yaml:"source_gate"`
    Status      string   `yaml:"status"` // active, dormant, stale, orphaned
    Paths       []string `yaml:"paths"`
    Tags        []string `yaml:"tags"`
    Pattern     string   `yaml:"pattern"`
    Recommend   string   `yaml:"recommendation"`
}
```

`signal.go` — Gate result signals (append-only, concurrent-safe):
```go
type Signal struct {
    Learning  string `yaml:"learning"`
    Branch    string `yaml:"branch"`
    GateID    string `yaml:"gate_id"`
    Result    string `yaml:"result"` // pass, fail, retry
    Timestamp string `yaml:"timestamp"`
    Files     []string `yaml:"files_touched"`
}
```

`store.go` — File-based storage:
- `Store.SaveLearning(l Learning)` → writes to `.belayer/.internal/learnings/{id}.yaml`
- `Store.SaveSignal(s Signal)` → writes to `.belayer/.internal/signals/{timestamp}-{branch}-{learning}-{result}.yaml`
- `Store.ListLearnings()` → reads all learning files
- `Store.ListSignals(learningID)` → reads signals for a learning

`check.go` — Retrieval:
- `Check(files []string) []Learning` — Tier 1: path match against learning paths. Tier 2: tag match. Return top N relevant, active learnings.
- `FormatContext(learnings []Learning) string` — Render as markdown for prompt injection.

`record.go` — Learning extraction:
- `RecordFromGate(gateResult, rationale, changedFiles) Learning` — Sequence: (1) calls cheap model to extract pattern, (2) generates UUID for learning ID, (3) calls `Store.SaveLearning()`, (4) calls `Store.SaveSignal()` with the new learning ID and result="fail". Returns the saved Learning.
- Learning IDs are UUIDs (avoid collision across concurrent branches). Deduplication of semantically similar learnings is deferred to post-MVP.
- Requires `claude` CLI in `$PATH`. If unavailable, `--skip-extraction` flag saves raw gate output as the pattern text without model structuring.

**Extraction prompt spec:** The model receives: (a) gate-result.json summary (dimension scores, issues), (b) rationale.md content, (c) diff-stat file list. It is asked to return JSON: `{"pattern": "...", "recommendation": "...", "paths": ["glob patterns"], "tags": ["category tags"]}`. The response is parsed into a `Learning` struct. If parsing fails, the raw rationale is saved as `pattern` with paths derived from the diff-stat.

`stats.go` — Analytics:
- `Stats() StatsReport` — For each learning: signal count, pass/fail ratio, last triggered. MVP status is always `active`. Status computation rules (dormant after N consecutive passes, stale after 90 days no signals, orphaned when paths deleted) are defined but implemented post-MVP.

**New CLI commands: `internal/cli/quality.go`**

```
belayer quality check --files src/auth/routes.ts src/middleware/handler.ts
belayer quality record --gate-id <taskID>-<nodeName>-<attempt> [--work-dir .]
belayer quality stats
```

`check` reads learnings, matches by path, outputs relevant patterns as markdown.
`record` resolves the gate-id to `{workDir}/.belayer/.internal/output/gate-result.json` and `rationale.md`. Accepts `--work-dir` override (defaults to cwd). Accepts `--skip-extraction` to bypass model call.
`stats` computes and displays analytics.

### Phase 2: Integration with Pipeline Engine

**Modify `internal/temporal/activity.go`:**

In `buildInputPrompt()`, after assembling the base prompt: parse `diff-stat.txt` (already materialized by `materializeCodeInput`) to extract file paths, call `quality.Check(files)`, and if learnings are returned, append `quality.FormatContext(learnings)` to the input prompt. This is the canonical injection point. Framework scripts should NOT also call `belayer quality check` manually (avoid double-injection).

On gate FAIL/RETRY in `processGateResult()`, automatically call `quality.RecordFromGate()` to capture the learning. The `changedFiles` list must be threaded from `materializeCodeInput` through `NodeActivity` into `processGateResult` (currently not passed, requires extending the function signature). This makes the flywheel automatic, no manual intervention needed.

**Modify signal emission:**

After every gate result (pass or fail), emit a signal with `quality.SaveSignal()`. This feeds the recurrence tracking. Pass signals are emitted independently of `RecordFromGate` (which handles fail signals). In `processGateResult`, on `outcome == Pass`, call `SaveSignal` with the learning IDs from the most recent `Check` result for this node's files. If no learnings were matched, no pass signal is emitted. For file list resolution, `processGateResult` re-parses `diff-stat.txt` from `{workDir}/.belayer/.internal/input/` rather than threading the list through the call chain.

**Path matching spec:** Learning `paths` are directory-level prefixes (e.g., `src/auth/`, `internal/gate/`), not glob patterns. The extraction prompt requests prefixes, not globs. Matching uses `strings.HasPrefix` on changed file paths. Full glob support deferred to post-MVP.

### Phase 3: claude-codex Framework

**New framework: `frameworks/claude-codex/`**

`framework.yaml` — metadata (name: claude-codex, version: 0.1.0)

`pipeline.yaml`:
```yaml
name: claude-codex
nodes:
  - name: implement
    type: node
    command: "./scripts/implement.sh"
    description: "Implement the design specification"
    output:
      type: commit
    on_pass: review
    on_fail: stop

  - name: review
    type: gate
    command: "./scripts/review.sh"
    description: "Adversarial code review"
    output:
      type: gate_result
    dimensions:
      - name: correctness
        weight: 0.30
      - name: edge_cases
        weight: 0.25
      - name: security
        weight: 0.20
      - name: test_coverage
        weight: 0.25
    thresholds:
      pass: 7.5
      retry: 4.0
    on_pass: stop
    on_retry: implement
    on_fail: stop
    max_retries: 2
```

`scripts/implement.sh` — Reads node-context.json, runs Claude with the description + input prompt. Learnings arrive pre-injected via the engine's `buildInputPrompt()` into `node-context.json`'s `InputPrompt` field. The script does NOT call `belayer quality check` separately (avoid double-injection).

`scripts/review.sh` — Reads node-context.json, runs Codex (or `claude -p` with a different model) in adversarial mode. Completely isolated, no implementation context. Only sees the diff.

Key difference from claude-tmux: implement uses Claude, review uses Codex. Two different models, intentionally isolated. H-as-feature.

## Open Questions

1. **Model for learning extraction:** Should `belayer quality record` call `claude -p --model haiku` or accept a configurable model? Haiku is cheapest but may not extract patterns well enough. MVP: hardcode haiku, add `--model` flag later.
2. **Micro-gates (Codex's idea):** Post-MVP. Should learnings eventually become additional gate checks? Would require learnings to carry an optional "verification question" field. Architecturally interesting but not in scope.
3. **Universal learnings:** Post-MVP. Global learnings (paths: ["*"]) that apply to all runs. Bootstrapping mechanism TBD.
4. **Learning deduplication:** Post-MVP. If same pattern extracted from two gate failures, duplicates accumulate. Model-assisted dedup later.
5. **Framework-level vs repo-level learnings:** Post-MVP. MVP: all learnings are repo-level. Framework scoping later if needed.

## Success Criteria

1. `belayer quality check --files <paths>` returns relevant learnings in < 500ms (file I/O only, no model call)
2. `belayer quality record --gate-id <id>` extracts a structured learning from gate failure output
3. Gate failures in the claude-codex pipeline automatically create learnings
4. Gate pass rate measurably improves after 10+ learning cycle (can be measured via `belayer quality stats`)
5. Concurrent branches don't corrupt learning or signal state
6. All quality commands have test coverage

## Distribution Plan

Belayer is distributed as a Go binary via `go install` and GitHub Releases. The claude-codex framework is embedded at compile time via `//go:embed`. No new distribution mechanism needed.

## Next Steps

2. **Add CLI commands** — `belayer quality check`, `belayer quality record`, `belayer quality stats` in `internal/cli/quality.go`
3. **Wire into pipeline engine** — Auto-inject learnings into implementation prompts via `buildInputPrompt()`. Auto-record learnings on gate failure via `processGateResult()`.
4. **Write tests** — Unit tests for store, check, record, signal. Integration test for the full capture-retrieve cycle.
5. **Create `frameworks/claude-codex/`** — Pipeline YAML + implement.sh + review.sh. Embed in Go binary.
6. **Smoke test** — Run the claude-codex framework on a real task, verify the learning loop fires.
