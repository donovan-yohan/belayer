# Agent-Friendly Architecture: Setter, Spotter, and Lead Redesign

**Date**: 2026-03-07
**Status**: Proposed

## Problem Statement

The current belayer architecture spawns Claude sessions from within Go code (`claude -p`), creating Claude-in-Claude nesting when belayer itself is called from an agent session. The CLI has interactive elements (brainstorm Q&A) that break non-interactive usage. The coordinator is tightly coupled to Claude-specific invocation patterns, making vendor swaps difficult.

### Goals

1. Eliminate Claude-in-Claude nesting
2. Remove all interactive functionality from the CLI
3. Make the CLI a pure data publisher (events to SQLite)
4. Enable vendor-agnostic agent execution (Claude, Codex, etc.)
5. Support parallel per-goal execution within repos
6. Provide cross-repo validation with automatic redistribution of followup work

## Architecture Overview

```
belayer (CLI binary)
|-- belayer manage    -> interactive agent session (the belayer)
|-- belayer setter    -> daemon per instance (the setter)
|   |-- lead sessions -> tmux windows per goal
|   +-- spotter       -> tmux window for cross-repo review
+-- belayer task/tui  -> existing commands
```

### Climbing Nomenclature

| Role | Climbing Term | Responsibility |
|------|--------------|----------------|
| Interactive CLI manager | **Belayer** (`belayer manage`) | Human-facing interface; interactive agent session for task creation, brainstorming, Jira ticket fetching |
| Daemon / process manager | **Setter** (`belayer setter`) | Designs execution paths; DAG executor that polls SQLite, spawns tmux sessions, manages lifecycle |
| Cross-repo reviewer | **Spotter** | Watches for cross-repo misalignment; reviews all diffs when leads complete |
| Per-goal worker | **Lead** | Goes first on the sharp end; one Claude Code session per goal running the harness flow |

## Components

### 1. Belayer (Interactive Manager)

`belayer manage --instance <name>`

An interactive agent session (Claude Code or Codex) trained on using the belayer CLI and performing brainstorming or Jira ticket fetching to create tasks.

**Responsibilities:**
- Convert Jira tickets to task specs (`spec.md` + `goals.json`)
- Run brainstorming skill to enrich task descriptions
- Invoke `belayer task create --spec spec.md --goals goals.json` to publish tasks
- Human-in-the-loop interface for the system

**Key principle:** All LLM inference for task creation happens here, not in the CLI or daemon. The CLI is a pure data publisher.

### 2. CLI (Pure Data Publisher)

The CLI becomes non-interactive. No internal Claude spawning. It validates input and writes to SQLite.

```bash
# Create a task from pre-generated spec and goals
belayer task create --instance foo --spec spec.md --goals goals.json

# Optional: create from Jira (just stores ticket ref, no LLM call)
belayer task create --instance foo --jira PROJ-123 --spec spec.md --goals goals.json
```

**Required inputs:**
- `--spec` (spec.md) — the full task specification / PRD
- `--goals` (goals.json) — per-repo goal decomposition

**Optional inputs:**
- `--jira` — Jira ticket reference for traceability
- `--instance` — target instance (defaults to default)

**What gets removed:**
- Sufficiency check (moved to belayer manage / brainstorm skill)
- Interactive brainstorm Q&A
- All `claude -p` invocations from CLI code

### 3. Task Format

#### spec.md

Rich PRD generated by the brainstorm skill or belayer manage session. Contains full context, requirements, constraints, and acceptance criteria.

#### goals.json

Per-repo goal decomposition. Tasks arrive fully decomposed — the setter does no inference, only routing.

```json
{
  "repos": {
    "api": {
      "goals": [
        {
          "id": "api-1",
          "description": "Add /users endpoint with role field",
          "depends_on": []
        },
        {
          "id": "api-2",
          "description": "Add integration tests for user roles",
          "depends_on": ["api-1"]
        }
      ]
    },
    "app": {
      "goals": [
        {
          "id": "app-1",
          "description": "Add user role display to profile page",
          "depends_on": []
        }
      ]
    }
  }
}
```

Goals within a repo can have dependencies (`depends_on`). Cross-repo dependencies are handled by the spotter at review time, not by explicit DAG edges — keeps the format simple and avoids over-specification.

### 4. Setter (Daemon)

`belayer setter --instance <name>`

A Go daemon that runs one per instance. Polls SQLite for new tasks and manages the full lifecycle.

**Core loop:**

1. Poll SQLite for tasks in `pending` status
2. Parse `goals.json` — build per-repo goal DAG
3. Create worktrees for each repo (existing worktree logic)
4. Spawn tmux session for the task (`task-<id>`)
5. For each repo, identify goals with no unmet dependencies
6. Spawn a tmux window per ready goal, launching a lead session
7. Watch for DONE markers per goal (file-based: `DONE` in worktree)
8. When a goal completes, check if it unblocks other goals — spawn those
9. When all goals for all repos complete, spawn a spotter session
10. Spotter approves — create PRs, mark task complete
11. Spotter rejects — write new per-repo correction goals, repeat from step 5
12. Max 2 spotter reviews — if still failing, mark repo as stuck

**tmux layout:**

```
tmux session: task-<id>
  +-- window: setter (coordinator pane, logs)
  +-- window: api-goal-1 (lead session)
  +-- window: api-goal-2 (lead session, spawned after api-goal-1 completes)
  +-- window: app-goal-1 (lead session, parallel with api-goal-1)
  +-- window: spotter (spawned when all leads complete)
```

**State tracking:**

All state lives in SQLite:
- Task status: `pending -> running -> reviewing -> complete/stuck`
- Goal status: `pending -> running -> complete/failed`
- Events: goal started, goal completed, spotter spawned, review verdict, PR created
- DONE markers: file-based for vendor agnosticism, mirrored to SQLite events by setter

### 5. Lead (Per-Goal Worker)

Each lead is a vanilla Claude Code session (or Codex, or any agent) launched in a tmux window, pointed at the correct worktree, with a prompt containing:

- The spec.md content
- The specific goal description
- Instructions to run the harness flow: `/harness:plan` -> `/harness:orchestrate` -> `/harness:complete`
- Instructions to write a `DONE` file when complete

**Key properties:**
- No custom lead.sh script — the agent uses existing harness skills
- No belayer-specific tooling inside the session
- Vendor-agnostic: swap the agent binary and prompt format, everything else stays the same
- One session per goal enables parallel execution of independent goals
- Clean context per goal — no pollution from prior goals

**Spawn interface (3 methods):**

```go
type AgentSpawner interface {
    // Spawn launches an agent in the given directory with the given prompt.
    // Returns a handle for monitoring.
    Spawn(ctx context.Context, workdir string, prompt string) (AgentHandle, error)
}

type AgentHandle interface {
    // Done returns a channel that closes when the agent exits.
    Done() <-chan struct{}
    // Kill terminates the agent.
    Kill() error
}

type DoneSignaler interface {
    // WatchForDone watches for a completion signal from the agent.
    WatchForDone(workdir string) <-chan struct{}
}
```

This is the vendor abstraction boundary. Claude today, Codex tomorrow.

### 6. Spotter (Cross-Repo Reviewer)

Spawned by the setter when all leads for a task complete. Runs as a Claude Code session (or equivalent) in its own tmux window.

**Input:**
- Original spec.md
- Git diffs from all repo worktrees
- Goal completion status

**Output — review verdict:**

```json
{
  "verdict": "approve" | "reject",
  "repos": {
    "api": {
      "status": "pass" | "fail",
      "goals": [
        "Fix the response schema to match frontend expectations"
      ]
    }
  }
}
```

**Behavior:**
- `approve` — setter creates PRs for all repos, marks task complete
- `reject` — setter writes correction goals for failing repos, spawns new lead sessions
- Max 2 review cycles. If still failing after 2 spotter reviews, mark affected repos as stuck and report to user

**Partial completion:** All PRs are held until the spotter approves. Cross-repo consistency is the whole point of belayer — partial PRs defeat the purpose.

## Data Flow

```
User / belayer manage
    |
    | (brainstorm skill, Jira fetch)
    v
belayer task create --spec spec.md --goals goals.json
    |
    | (write to SQLite)
    v
SQLite: task pending
    |
    | (setter polls)
    v
Setter: parse goals, create worktrees, build DAG
    |
    | (spawn tmux windows)
    v
Leads: harness:plan -> orchestrate -> complete -> DONE file
    |
    | (setter watches DONE files, updates SQLite)
    v
All goals complete -> Setter spawns Spotter
    |
    v
Spotter: review diffs against spec
    |
    +-- approve -> Setter creates PRs -> task complete
    |
    +-- reject -> Setter writes correction goals -> new leads
                  (max 2 review cycles, then stuck)
```

## SQLite Schema Changes

New/modified tables needed:

```sql
-- Goals table (new)
CREATE TABLE goals (
    id TEXT PRIMARY KEY,
    task_id TEXT NOT NULL REFERENCES tasks(id),
    repo_name TEXT NOT NULL,
    description TEXT NOT NULL,
    depends_on TEXT, -- JSON array of goal IDs
    status TEXT NOT NULL DEFAULT 'pending',
    attempt INTEGER NOT NULL DEFAULT 0,
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP,
    completed_at DATETIME
);

-- Spotter reviews (new)
CREATE TABLE spotter_reviews (
    id INTEGER PRIMARY KEY AUTOINCREMENT,
    task_id TEXT NOT NULL REFERENCES tasks(id),
    attempt INTEGER NOT NULL,
    verdict TEXT NOT NULL, -- 'approve' or 'reject'
    output TEXT, -- full review JSON
    created_at DATETIME DEFAULT CURRENT_TIMESTAMP
);
```

Existing tables (`tasks`, `events`, `leads`) remain. The `leads` table may be simplified since leads no longer have the custom verdict loop — they just run and produce a DONE marker.

## Migration Path

This is a significant redesign. Suggested phases:

### Phase 1: CLI cleanup
- Remove interactive brainstorm from CLI
- Remove `claude -p` calls from task.go
- Add `--spec` and `--goals` flags to `task create`
- CLI becomes pure data publisher

### Phase 2: Setter daemon
- Implement `belayer setter --instance <name>`
- Goal DAG parsing and execution
- tmux session/window management
- DONE file watching
- SQLite event tracking

### Phase 3: Lead spawning
- Implement `AgentSpawner` interface
- Claude Code implementation (tmux + claude with prompt)
- Per-goal session management
- Harness flow prompt template

### Phase 4: Spotter
- Cross-repo review agent
- Verdict parsing and redistribution
- Max 2 review cycles with stuck reporting
- PR creation on approval

### Phase 5: Belayer manage
- Interactive agent session for task creation
- Brainstorm skill integration
- Jira ticket fetching and conversion

## Design Decisions

| Decision | Choice | Rationale |
|----------|--------|-----------|
| Process management | tmux | Battle-tested, gives visibility, no custom daemon complexity |
| Agent communication | DONE files + SQLite events | Vendor-agnostic; any agent can write a file |
| Goal granularity | One session per goal | Parallelism, isolation, clean context, retry granularity |
| Cross-repo deps | Implicit (spotter validates) | Explicit cross-repo DAG edges are fragile and over-specified |
| Max spotter reviews | 2 | Prevents infinite loops; stuck repos reported to user |
| Partial completion | Hold all PRs | Cross-repo consistency is belayer's core value proposition |
| Vendor abstraction | AgentSpawner interface | 3-method boundary: spawn, done, kill |
| Task decomposition | Pre-decomposed by user/skill | Setter does no inference, only routing |
| TUI integration | Hybrid: SQLite dashboard + tmux attach | Dashboard for overview, tmux for live agent output |
| Crash recovery | DONE files as source of truth | Setter reconstructs state from worktree DONE files + SQLite on restart |
| Output capture | Structured DONE.json + tmux pipe-pane logs | DONE.json for spotter/audit, pipe-pane for debugging with log rotation/cleanup |
| Concurrent tasks | Single event loop with lead queue | `--max-leads` caps concurrency (default 8), FIFO queue for overflow |

## Resolved Design Questions

### 1. TUI Integration — Hybrid (SQLite + tmux attach)

The TUI serves two complementary needs:

**Dashboard view (SQLite-backed):** The existing bubbletea TUI evolves to show goal-level detail. The hierarchy becomes tasks -> goals (grouped by repo) -> events. Left pane lists tasks, middle pane shows goals with status (pending/running/complete/stuck), events pane shows goal-level events. Polls SQLite like today.

**Live view (tmux attach):** Pressing `enter` on a goal in the TUI attaches to that goal's tmux window, showing real-time agent output. This gives full visibility into what a lead is doing without parsing logs.

### 2. Setter Crash Recovery — DONE Files as Source of Truth

Recovery flow when the setter restarts:

1. Read all tasks in `running` status from SQLite
2. For each running task, check tmux session existence (`tmux has-session -t task-<id>`)
3. **Session alive, windows alive**: Re-attach DONE file watchers. Resume DAG.
4. **Session alive, some windows dead**: Check for DONE files in worktrees of goals marked `running`. If DONE file present, mark complete. If absent, mark failed, increment attempt counter, re-spawn if under retry limit.
5. **Session gone**: Scan all worktrees for DONE files. Update goal statuses accordingly. Re-create tmux session if goals remain.

**Stale detection:** Goals in `running` status whose tmux window is dead and have no DONE file are marked failed after a configurable timeout (default 30 minutes since last status change).

**SQLite stores for recovery:**
- Goal status with attempt counter
- tmux session/window names
- Timestamp of last status change (for stale detection)

### 3. Agent Output Capture — Structured Results + tmux Pipe Logging

Both mechanisms are used:

**Structured results (primary):** Leads write a `DONE.json` file containing structured output:

```json
{
  "status": "complete",
  "summary": "Added /users endpoint with role field",
  "files_changed": ["api/handlers/users.go", "api/handlers/users_test.go"],
  "notes": "Used existing auth middleware pattern"
}
```

The setter reads `DONE.json`, stores the summary in SQLite events, and passes it to the spotter for review context.

**tmux pipe-pane logging (debug/audit):** The setter enables `tmux pipe-pane` on every lead window at spawn time, capturing full terminal output to a log file:

```bash
tmux pipe-pane -t <window> "cat >> <log_dir>/task-<id>/<goal-id>.log"
```

This captures output even when a window or pane crashes/exits unexpectedly, since pipe-pane writes continuously to the file.

**Log management:** Logs can grow large with verbose agent output. The setter includes log lifecycle management:

- **Max log size per goal:** Configurable limit (default 10MB). When exceeded, the log is rotated — the oldest half is truncated and a `[truncated]` marker is inserted.
- **Max total log size per instance:** Configurable limit (default 500MB). When exceeded, oldest task logs are compressed (gzip) or purged.
- **Auto-cleanup on task completion:** When a task reaches `complete` or `stuck` status, its logs are compressed. Compressed logs are retained for a configurable duration (default 7 days), then deleted.
- **CLI commands for log management:**
  - `belayer logs --task <id>` — view/tail logs for a task
  - `belayer logs --goal <id>` — view a specific goal's log
  - `belayer logs cleanup` — manually trigger log cleanup
  - `belayer logs stats` — show log disk usage per instance/task

### 4. Concurrent Tasks — Single Event Loop with Lead Queue

The setter handles multiple tasks concurrently via a single event loop. Each task is an independent state machine.

**Event loop:**

```
1. Poll for new pending tasks -> initialize them
2. For each running task:
   a. Check for newly completed goals -> update DAG, queue unblocked goals
   b. Check for stale goals -> handle timeouts
   c. Check if all goals complete -> spawn spotter
   d. Check for spotter verdict -> approve/redistribute/stuck
3. Process lead queue -> spawn leads up to max concurrency
```

**Resource limits:** A configurable `--max-leads` flag (default 8) caps the total number of concurrent lead sessions across all tasks. Goals that are ready but can't be spawned due to the cap enter a FIFO queue. The setter spawns queued goals as running leads complete.

**Isolation:** Each task gets its own tmux session and worktrees. No shared state between tasks beyond the SQLite database and the lead concurrency pool.
