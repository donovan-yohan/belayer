# Design: Self-Improving Harness Runtime (.harness/)

Generated by /office-hours on 2026-03-28
Branch: dy/feat/harness-plan-lifecycle-improvements
Repo: donovan-yohan/belayer
Status: APPROVED
Mode: Builder

## Problem Statement

The harness plugin (v4.1.0) captures learnings and enforces them during review, but the agent definitions themselves are static markdown. When `/harness:review` misses a bug class, a human must manually edit the command definition or REVIEW_GUIDANCE.md. The mechanism responsible for improvement is not itself improvable.

Three gaps compound this:
1. **No self-modification loop.** Agent definitions never evolve based on what they catch vs. miss.
2. **No quantitative feedback.** No data on which review agents catch real issues, how often plans drift, or whether learnings prevent recurrence. Without metrics, you cannot distinguish load-bearing components from ceremony.
3. **No QA or documentation layer.** The pipeline goes brainstorm -> plan -> orchestrate -> review -> reflect -> complete, but skips visual QA, design review, and structured documentation updates.

Informed by: Anthropic's "Harness design for long-running application development" (GAN-inspired generator/evaluator separation, context resets, simplification principle) and Meta's HyperAgents paper (self-referential agents where the improvement mechanism itself is improvable, persistent memory, open-ended variant archives, cross-domain meta-transfer).

## What Makes This Cool

You run the harness on a repo for a few weeks. Then you open `.harness/agents/reviewer.md` and see that the reviewer learned to check for connection pool exhaustion because it missed one in week 3. You open `.harness/metrics/review-effectiveness.json` and see that the comment-analyzer agent has a 83% false positive rate and 0 unique catches, so the harness disabled it for this repo. You open `.harness/memory/IMPROVEMENTS.md` and read the agent's own reasoning for why it changed its prompts.

Then you look at the learnings. Some are tagged `scope: repo` ("this codebase has race conditions in the pipeline engine"). Others are tagged `scope: universal` ("review agents should always check stderr capture in shell commands"). The universal ones, if you've opted in, get packaged as GitHub issues against the harness plugin repo with the data and reasoning attached. The community of users becomes a distributed improvement system feeding back into the open source project.

Any LLM can read the `.harness/` folder. Switch from Claude to Codex to Gemini, your repo's accumulated intelligence persists. The repo owns its ways of working. The AI tool is swappable.

## Constraints

- Plugin-first: self-improvement lives in the harness plugin, not in belayer Go code. Belayer integration (replay/eval) is a future composition layer.
- Model-agnostic: `.harness/` uses plain files (markdown, JSON, YAML) that any LLM can read and write. No proprietary formats, no runtime dependencies.
- Backward compatible: repos without `.harness/` work exactly as today. The plugin checks `.harness/` first, falls back to `plugins/harness/` defaults.
- Two storage tiers: `.harness/` in-repo (committed, shared with team) and `~/.harness/repo-slug/` global (personal, not committed). Users who are demoing or not ready to commit use the global path.
- Merge-friendly: all formats are append-only or use stable keys. No sequential numbering. Date-stamped entries.

## Premises

1. Agent definitions (markdown prompts) are the right unit of self-modification, not code, not config, not weights.
2. Per-repo `.harness/` (with `~/.harness/repo-slug/` global fallback) is the right storage layer.
3. The reflect phase is the right moment for self-modification, with full context of what happened, what was caught, and what was missed.
4. Metrics are necessary to distinguish load-bearing components from ceremony. Without data, evolution is random.
5. Local-vs-universal improvement classification is feasible. The harness can distinguish "this repo has race conditions" from "review agents should always check stderr capture."
6. Community feedback via GitHub issues (opt-in) is the right mechanism for universal improvements to flow back to the open source project.
7. The `.harness/` protocol can be model-agnostic. Any LLM can read agent definitions, write metrics, and propose agent evolution.

## Cross-Model Perspective

### Codex (GPT-5.4)

Proposed a **harness evolution league**: every reflect produces a candidate patch, then Belayer runs baseline vs. mutated harnesses on a held-out replay set. Winners get promoted; losers stay local. Identified Microsoft's APM as the closest existing tool (repo-native agent config), with the self-improving control loop as the missing 50%. Weekend build: `.harness/manifest.yaml` + `belayer harness reflect` as smallest closed loop.

### Claude subagent (Opus 4.6)

Proposed **adversarial co-evolution**: fork the harness, run both instances on the same spec via belayer's setter/spotter, score which produced better outcomes. Identified Mem0 as the closest existing tool (memory layer with deduplication), with classification, action, and pipeline integration as the missing 50%. Key distinction: "Mem0 is a service, the harness is a protocol." Weekend build: `/harness:evolve` command + `harness-evolver` agent + scope classification in learnings format.

### Synthesis

Both agree repo-owned model-agnostic intelligence is the center of gravity. Both agree the weekend build is classification + evolution. Codex pushes toward replay/eval (test proposals before promoting). Claude pushes toward adversarial co-evolution (two harnesses competing). Both are future layers that compose on top of the core: get the classification and evolution loop working first, then add replay and adversarial modes.

## Approaches Considered

### Approach A: Incremental Evolution (Minimal Viable)
Add `scope: repo | universal` to learnings format, a `/harness:evolve` command, and `~/.harness/` as global fallback. No metrics, no QA, no new agents. Ships in days.

### Approach B: Metrics-Driven Self-Improvement (Full Runtime) -- CHOSEN
Full `.harness/` runtime. Agent definitions evolve. Metrics track review effectiveness, plan drift, learning recurrence. `/harness:evolve` classifies, proposes, records. `/harness:test` for QA + design review. `/harness:document` for release docs. Data-driven self-improvement.

### Approach C: Community-Aware Protocol (Spec-First)
Define `.harness/` as a format spec with `manifest.yaml`. Any tool can read/write. Community evolution via shared registry. Highest ceiling, but premature abstraction risk.

## Recommended Approach

Approach B: Metrics-Driven Self-Improvement. Build the full `.harness/` runtime with metrics, agent evolution, QA, and documentation. Let the protocol emerge from usage, not from spec design.

## .harness/ Directory Structure

```
.harness/                              # per-repo adaptive runtime (or ~/.harness/repo-slug/)
|-- manifest.yaml                      # repo identity, harness version, feature flags
|-- config.yaml                        # repo-specific tuning (review thresholds, agent toggles, contribute opt-in)
|-- agents/                            # evolved agent definitions (started from plugin defaults)
|   |-- reviewer.md                    # repo-specific review agent
|   |-- planner.md                     # repo-specific planning agent
|   |-- evaluator.md                   # independent evaluator (Anthropic article's GAN pattern)
|   |-- debugger.md                    # repo-specific debugging agent
|   `-- evolver.md                     # meta-agent that proposes modifications to other agents
|-- metrics/                           # quantitative self-assessment
|   |-- review-effectiveness.json      # per-agent: runs, findings, false positives, unique catches
|   |-- plan-accuracy.json             # drift frequency, surprise frequency, completion rate
|   |-- learning-efficacy.json         # recurrence rate for addressed bug classes
|   |-- test-effectiveness.json        # per-test-type: runs, findings by severity, false positives
|   `-- phase-costs.json               # tokens, time, iterations per phase
|-- memory/                            # persistent memory across sessions
|   |-- IMPROVEMENTS.md                # audit trail of all self-modifications with reasoning
|   `-- session-history.json           # timestamped session summaries for cross-session context
|-- proposals/                         # pending agent evolution proposals
|   `-- {YYYY-MM-DD}-{slug}.md        # proposed edit to an agent definition, with diff and reasoning
|-- handoffs/                          # context reset artifacts for long-running sessions
|   `-- {session-id}.json             # structured state for context handoff between windows
`-- .gitignore                         # ignore handoffs/, session-history.json (ephemeral)
```

## New Commands

### /harness:init (extended)

Add `.harness/` scaffold alongside `docs/` scaffold. Copy default agent definitions from `plugins/harness/agents/` to `.harness/agents/`. Initialize empty metrics files. Create `manifest.yaml` with harness version and creation date. Create `config.yaml` with sensible defaults.

If user specifies `--global`, scaffold to `~/.harness/repo-slug/` instead.

### /harness:evolve (NEW)

Runs after reflect, before complete. The self-modification phase.

**Phase 1: Scope Classification**
- Read all learnings written in this session (from `docs/LEARNINGS.md`, filter by current date/branch)
- For each learning, classify as `scope: repo` or `scope: universal`:
  - Default to `repo` (conservative). Only promote to `universal` when ALL of:
    - Contains zero project-specific references (file paths, module names, domain concepts)
    - Matches a known general pattern category (error handling, testing strategy, review methodology, agent coordination, security, performance)
    - The recommendation is actionable without project-specific context
  - `repo`: references specific file paths, module names, project-specific concepts, or domain-specific patterns
  - `universal`: describes general patterns without project-specific references
  - Classification can be overridden by user during proposal review in `/harness:evolve` Phase 4
- Write scope tag into learning entry as new metadata field

**Phase 2: Metrics Update**
- **Prerequisite:** `/harness:review` must write a structured `review-results.json` to `.harness/` listing each agent's findings with structured metadata (finding text, severity, accepted/dismissed, unique). This is a new output contract for the review command.
- Read `.harness/metrics/review-effectiveness.json` and `.harness/review-results.json`
- For each review agent that ran in this session (from review-results.json):
  - Increment `runs`
  - Count findings that led to actual code changes (`findings`)
  - Count findings that were dismissed or reverted (`false_positives`)
  - Count findings no other agent caught (`unique_catches`)
- Update `plan-accuracy.json` with drift/surprise counts from the living plan
- Update `learning-efficacy.json`: check if any bugs in this session match existing learning categories (recurrence detection)
- Update `phase-costs.json` with timing data. **Data source:** each harness command records `start_time` and `end_time` timestamps to `.harness/phase-timing.tmp` (a transient file, overwritten per phase). Evolve reads these to compute duration. Token counts are best-effort: derived from Claude API response `usage` fields if available in conversation context, otherwise omitted. Token tracking is informational, not load-bearing.

**Phase 3: Agent Evolution Proposals**
- Read `.harness/metrics/review-effectiveness.json`
- Read review escapes from this session (from `docs/REVIEW_GUIDANCE.md` escape log)
- Read `scope: universal` learnings from this session
- For each signal (escape, metric anomaly, universal learning):
  - Read the relevant agent definition from `.harness/agents/`
  - Propose a concrete edit (add a check, modify a threshold, add a question to the prompt)
  - Write proposal to `.harness/proposals/{YYYY-MM-DD}-{slug}.md` with:
    - Signal source (escape ID, metric, learning ID)
    - Current agent text (relevant section)
    - Proposed change (diff format)
    - Reasoning (why this change should improve outcomes)
    - Classification: `scope: repo` or `scope: universal`

**Phase 4: Auto-Apply Safe Proposals**
- Proposals that meet ALL of these criteria are auto-applied:
  - The signal is a review escape (concrete evidence of a miss)
  - The change is additive (adds a check, doesn't remove one)
  - The agent has run N+ times where N = `config.yaml` `evolve.min_runs_for_auto` (default: 5)
- All other proposals require human review (surfaced in PR description)
- Applied changes are logged to `.harness/memory/IMPROVEMENTS.md`

**Phase 5: Community Contribution (opt-in) -- FUTURE MILESTONE**
- Deferred from core evolution loop. The classification, proposal, and auto-apply phases are the core. Community contribution is a distribution mechanism that ships separately.
- When implemented: if `config.yaml` has `contribute: true`, draft GitHub issues for `scope: universal` proposals against the harness plugin repo using `gh issue create` with label `community-evolution`. Gate behind user review before submission.

### /harness:test (NEW)

Runs after orchestrate, before review. The QA layer.

**Phase 1: Test Type Detection**
- Analyze the diff to determine what changed:
  - UI components changed? -> visual QA (`/qa` or `/qa-only` pattern)
  - Design system touched? -> design review (`/design-review` pattern)
  - API endpoints changed? -> API smoke tests
  - Config/infra changed? -> smoke test the deploy path
- Only run the relevant test types

**Phase 2: Visual QA (if applicable)**
- If the project has a running dev server or static build:
  - Navigate to affected pages
  - Screenshot before/after
  - Check for visual regressions, layout breaks, responsive issues
  - Report findings with screenshots and repro steps
- Uses gstack `/browse` or Playwright MCP if available, falls back to manual checklist

**Phase 3: Design Review (if applicable)**
- If UI changes exist:
  - Check spacing consistency, typography hierarchy, color harmony
  - Check interaction states (loading, empty, error, success)
  - Check against DESIGN.md design system if it exists
  - Report findings with severity and suggested fixes

**Phase 4: Findings Report**
- Structured output: test type, findings, severity, screenshots/evidence
- Critical findings block progression to review
- Medium/low findings are noted for review phase

**Phase 5: Metrics**
- Write test results to `.harness/metrics/test-effectiveness.json`
- Track: tests run, findings by severity, false positives, time spent

### /harness:document (NEW)

Runs during complete, after reflect. The documentation release step. Inspired by gstack's `/document-release`.

**Phase 1: Diff-Driven Doc Audit**
- Determine the diff scope (same as reflect Phase 1)
- For each documentation file in the repo (README.md, ARCHITECTURE.md, DESIGN.md, CONTRIBUTING.md, CLAUDE.md, API docs):
  - Check if the diff contradicts, obsoletes, or gaps the doc
  - Flag specific sections that need updating
  - Distinguish: "this doc is wrong" (error) vs. "this doc is incomplete" (gap) vs. "this doc is fine" (pass)

**Phase 2: Auto-Updates**
- For errors and gaps, propose specific edits
- Apply safe auto-updates (fixing stale file paths, updating code examples that changed, removing references to deleted code)
- Flag risky updates for human review (changing architectural descriptions, modifying user-facing docs)

**Phase 3: CHANGELOG**
- If CHANGELOG.md exists:
  - Generate entry from diff and commit history
  - Match existing voice and format
  - Never clobber existing entries
- If no CHANGELOG.md, skip

**Phase 4: Cross-Doc Consistency**
- Check that all docs agree with each other
- Check that CLAUDE.md Documentation Map links are valid
- Check that README quickstart commands actually work (if testable)

**Phase 5: TODOS Cleanup**
- If docs/TODOS.md exists:
  - Mark completed items based on diff
  - Flag items that are now stale
  - Reorganize if needed

## Multi-Worktree Considerations

When multiple worktrees are active on the same repo:

1. **Shared `.harness/` state:** `.harness/` lives in the repo root, which is shared across worktrees via git. Worktrees on different branches may have different `.harness/` states.
2. **Metrics accumulation:** Metrics files use stable JSON keys with integer counters. Concurrent worktrees writing metrics can lose increments (two worktrees both read `runs: 5`, both write `runs: 6`, one increment lost). This is accepted as minor data loss for simplicity. Metrics are directional indicators, not precise accounting. For teams needing exact counts, use `~/.harness/` global mode (per-machine, no merge conflicts) or coordinate worktree timing.
3. **Proposal conflicts:** Two worktrees may propose conflicting agent modifications. Mitigation: proposals are dated files in `.harness/proposals/`, not direct edits. Conflicting proposals are both preserved and resolved during review.
4. **Global fallback isolation:** `~/.harness/repo-slug/` is per-machine, no merge conflicts. But no team sharing.
5. **Handoffs are per-worktree:** `.harness/handoffs/` is gitignored, so each worktree has its own handoff state.

## Context Isolation: Stateless Command Protocol

The Anthropic article's #1 finding: context coherence loss is the primary failure mode for long-running tasks, solved by full context resets with structured handoffs. The HyperAgents paper confirms: agents that switch roles in the same context bias toward their prior role's judgments.

**Architectural principle: every harness command is a stateless function.**

Each command reads its inputs from disk, does its work, writes its outputs to disk. The conversation window is the execution engine, not the state store. Fresh context per command. Files are the handoff.

This follows the same protocol as belayer nodes (`node-context.json` → work → completion files). The harness equivalent is `run-state.json`.

### run-state.json (ephemeral, gitignored)

Tracks the lifecycle of the current feature run:

```json
{
  "schema_version": 1,
  "plan": "docs/exec-plans/active/2026-03-28-self-improving-harness.md",
  "design_doc": "docs/design-docs/2026-03-28-harness-runtime-design.md",
  "branch": "dy/feat/harness-runtime",
  "phase": "review",
  "completed_phases": [
    { "name": "brainstorm", "completed_at": "2026-03-28T15:00:00Z" },
    { "name": "plan", "completed_at": "2026-03-28T15:30:00Z" },
    { "name": "orchestrate", "completed_at": "2026-03-28T17:00:00Z" }
  ],
  "started_at": "2026-03-28T15:00:00Z",
  "last_updated": "2026-03-28T18:30:00Z"
}
```

### Command contract

Every harness command:
1. **Reads** `run-state.json` on startup to know where it is in the lifecycle
2. **Reads** its specific inputs from disk (plan, learnings, metrics, review results, runs/)
3. **Does its work** with full context from files, not from conversation history
4. **Writes** outputs to disk (updated plan, new learnings, proposals, review results)
5. **Updates** `run-state.json` with phase completion

Commands are stateless functions. Users run `/clear` between each command (or belayer spawns separate sessions). Both paths produce identical results.

### Recommended user workflow

```
/harness:brainstorm   → writes design doc       → /clear
/harness:plan         → reads design doc, writes plan → /clear
/harness:orchestrate  → reads plan, writes code  → /clear
/harness:test         → reads diff, writes test results → /clear
/harness:review       → reads diff, writes review-results → /clear
/harness:reflect      → reads all outputs, writes learnings + evolve → /clear
/harness:document     → reads diff + docs, writes doc updates → /clear
/harness:complete     → reads plan + proposals, archives + PR
```

Each arrow is a context boundary. The run-state.json file connects them.

### Belayer pipeline composition (M2)

For autonomous execution, belayer can compose the same commands into a pipeline where each node is a fresh tmux session. The pipeline YAML reads the same run-state.json. This is additive, not required for M1.

### What gets removed

`/harness:loop` is deprecated. It was a compromise: running the full pipeline in a single session with shared context. The correct pattern is stateless commands with disk-based handoffs, whether invoked manually (user types each command in a fresh context) or autonomously (belayer pipeline).

### What stays

Individual commands remain as Claude Code skills. They work in any context: interactive sessions, belayer pipelines, Codex invocations, or any LLM that can read the plugin's command definitions. The run-state.json protocol makes them composable without coupling.

## Open Questions

1. **Metrics cold start:** How many sessions before metrics are meaningful? Should the harness explicitly mark early sessions as "learning period" and suppress evolution proposals?
2. **Agent definition drift:** If two team members evolve the same agent differently, how do conflicting evolutions merge? Git merge on markdown is workable but not great for prompt quality.
3. **Evaluation gaming:** If the evaluator criteria evolve based on what the evaluator catches, could the evaluator optimize for findable issues rather than important ones? **Decision: defer evaluator self-evolution until external signal sources (PR review feedback, production incidents) are integrated as ground truth.** The evaluator's criteria can be manually tuned, but auto-evolution of the evaluator is circular without external validation.
4. **Context reset protocol:** The Anthropic article identifies context coherence loss as the primary failure mode. Should `.harness/handoffs/` define a structured handoff format for context resets during long orchestration sessions?
5. **Global-to-repo migration:** When a user is ready to commit `~/.harness/repo-slug/` to the repo as `.harness/`, what's the migration path? Copy + git add? Or a `/harness:promote` command?
6. **Test infrastructure dependency:** `/harness:test` for visual QA needs a running dev server or browse binary. How does the harness detect availability and degrade gracefully?

## Persistence Layer: Deterministic Scripts

The harness plugin ships shell scripts that handle all deterministic file I/O. Commands invoke these scripts for the persistence layer, then do LLM-specific work (classification, proposal generation, reasoning). Scripts are testable independently without invoking any LLM.

**Scripts (in `plugins/harness/scripts/`):**

| Script | Purpose | Key parameters |
|--------|---------|---------------|
| `harness-init.sh` | Scaffold `.harness/` directory | `--harness-dir`, `--repo-slug`, `--global` |
| `harness-read-state.sh` | Read run-state.json to stdout | `--harness-dir` |
| `harness-update-state.sh` | Update run-state.json phase/completion | `--harness-dir`, `--phase`, `--plan` |
| `harness-write-metrics.sh` | Increment metric counters from run data | `--harness-dir`, `--agent`, `--findings`, `--false-pos` |
| `harness-write-proposal.sh` | Create proposal markdown file | `--harness-dir`, `--slug`, `--scope`, `--signal` |
| `harness-write-run.sh` | Write timestamped run record to runs/ | `--harness-dir`, `--phase`, `--branch`, `--data` |
| `harness-resolve-dir.sh` | Resolve .harness/ vs ~/.harness/slug/ | `--repo-root` (returns path or empty) |

Each script: reads parameters, does deterministic file operations, returns predictable output. Testable with standard shell assertions.

**Command/script separation:**
- Scripts handle: creating directories, writing JSON, updating run-state, reading configs, resolving storage tiers
- Commands handle: LLM judgment (scope classification, proposal generation, agent evolution reasoning)
- Commands invoke scripts via `bash plugins/harness/scripts/harness-*.sh`
