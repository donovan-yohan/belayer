# Design: Belayer v2 — Temporal-Backed Orchestrator Platform

Generated by /plan-ceo-review + /plan-eng-review on 2026-03-19
Branch: master | Mode: SCOPE EXPANSION
Repo: donovan-yohan/belayer

## Summary

Rewrite belayer from the ground up as a declarative, Temporal-backed orchestration platform. Users define pipeline topology via a DSL file, plug in role implementations (Go-native defaults or external commands), and belayer handles execution semantics. Each role has typed input/output contracts, risk-based autonomy gates, and independent eval/test support. This is a v2 clean break — not an incremental migration.

## Vision

### 10x Check
Instead of hardcoding roles in Go, belayer ships with a declarative pipeline DSL (YAML/TOML) that declares the topology — roles, connections, loops, providers, risk gates. Users fork the default pipeline, add/remove roles, swap implementations. The CLI becomes `belayer pipeline validate`, `belayer pipeline run`, `belayer pipeline show`. Belayer is "Terraform for AI agent pipelines."

### Platonic Ideal
You declare your pipeline. Each node has a typed contract (JSON Schema for input/output). You plug in implementations — any executable that speaks the contract, or use the Go-native defaults. Belayer handles execution semantics (retry, timeout, compensation, loops, fan-out/fan-in). Temporal provides the runtime. Users start from templates and customize. The Temporal Web UI gives full visibility. `belayer eval <role>` tests any role independently. Auto-recorded fixtures build eval suites over time. Risk-based autonomy gates let each role decide what passes through vs. what needs human review. Climbing metaphors throughout.

## Core Architecture Decisions

- **v2 clean break**: Not an incremental migration. Complete rewrite of the orchestration layer.
- **Temporal as backbone**: Temporal Go SDK for durable execution, messaging, and visibility
- **Language**: Go (Temporal's flagship SDK, single binary distribution)
- **Two provider contracts**: Type A (judgment calls, JSON in/out) and Type B (interactive sessions, CLI-callback)
- **Three pipeline phases**: Approach (intake) → Ascent (execution) → Send (output)
- **Provider pattern**: Go-native default roles, users override via provider config pointing to external commands. Same pattern as existing envprovider.
- **Pipeline topology**: Declarative DSL file. Belayer is opinionated about roles and their order within each phase. Users customize implementations per-role and choose depth per phase.
- **Temporal management**: `belayer temporal start/stop/status` manages the dev server lifecycle
- **Auth model**: Built-in roles shell out to `claude` CLI (uses the user's existing subscription). Frictionless — bring whatever coding agent, tool, or auth you have. API keys and subscriptions both work.
- **Temporal operational requirements**: Workflow versioning via `GetVersion()`, mandatory heartbeats for Type B roles (60s), file-based persistence on dev server

## Two Provider Contracts

### Type A — Pitches (Judgment Calls)
Short-lived, deterministic contract for roles that process structured data.

```
Input: JSON on stdin or --input flag
Process: command executes, produces result
Output: JSON on stdout
Duration: seconds to minutes
Examples: decomposer, spotter, anchor, pr-creator, pr-manager
```

### Type B — Ascents (Interactive Sessions)
Long-lived, opaque execution for roles that require interactive agent sessions.

```
Start: belayer spawns session (tmux, subprocess, etc.)
       with system prompt: "You are a <role>. When done, call
       `belayer <role> finish --task-id <id>`"
Run:   session works autonomously (research, code, test, review)
       uses whatever tools, skills, workflows the user has
End:   session calls: belayer <role> finish --task-id <id>
       (or: belayer <role> flare --task-id <id> for help)
       (or: belayer <role> fail --task-id <id> for graceful failure)
Safety: heartbeat every 60s; if session dies without callback → flare

CLI callback → Temporal Signal → workflow advances to next role
```

The session doesn't need to know about Temporal — it just calls a CLI command when done. This works with ANY interactive tool (Claude Code, Codex, Cursor, VS Code agents) as long as it can execute CLI commands.

## MVP Strategy

Prove the hardest part first: the pipes and the interactive session contract.

```
MVP: setter (Type B) → lead (Type B) via Temporal

  SETTER SESSION                         LEAD SESSION
  ──────────────                         ────────────
  User opens Claude Code                 Temporal triggers spawn
  with system prompt:                    with system prompt:
  "You are a setter. When your           "You are a lead. Execute
   plan is ready, call                    this plan. When done, call
   belayer setter finish"                 belayer lead finish"

  User works freely:               ───►  Agent works autonomously:
  research, brainstorm, plan      Temporal code, test, review
                                  Signal
  Calls: belayer setter finish           Calls: belayer lead finish
  → Temporal Signal sent                 → Temporal Signal sent
```

This proves: (a) interactive sessions work with CLI-callback, (b) Temporal pipeline delivers messages between roles.

## Safety Controls

| Control | Default | Configurable? |
|---------|---------|---------------|
| Max child work depth per role | 2 | Yes, in pipeline DSL |
| Global child budget per run | 50 items | Yes |
| Child work deduplication | Enabled (hash-based) | Yes |
| Risk gate approval expiry | 24 hours | Yes |
| Cancellation propagation | Parent cancel → all children cancel | Always (Temporal-native) |
| Activity heartbeat interval | 60 seconds (Type B only) | Yes |
| Max loop iterations (fall + re-attempt) | 3 | Yes, per loop in DSL |

## Accepted Scope

1. **Declarative Pipeline DSL** (YAML/TOML) — core abstraction
2. **Two Provider Contracts** — Type A (JSON in/out) and Type B (CLI-callback)
3. **Three Pipeline Phases** — Approach / Ascent / Send
4. **Eval Framework** — `belayer eval <role>` for independent role testing
5. **Risk-Based Autonomy Gates** — per-role configurable risk thresholds
6. **Pipeline Templates** — solo, team, research, enterprise
7. **Pipeline Visualization** — `belayer pipeline show` ASCII diagrams
8. **Fixture Recording** — auto-snapshot role I/O as eval fixtures
9. **Pipeline Slicing** — `--from`/`--to` flags for partial runs
10. **Self-Generating Work Tickets** — any role can emit child work items
11. **Batteries-Included Implementations** — existing code restructured as defaults
12. **Safety Controls** — depth limits, budgets, dedupe, expiry, cancellation
13. **Temporal Operational Requirements** — versioning, heartbeats, persistence

## Key Constraints

- v2 clean break — not incremental migration from v1
- "When belayer is operating at full force, it should be able to meaningfully create its own work tickets, with each layer having the ability to control a 'risk criteria' for what is autonomously passed through vs. raised for a human to review."
- "Keep the climbing metaphors for our core"
- "Focus on making it frictionless for people to bring the code-review agent or tool they like into belayer, or use whatever coding agent or workflow they have, whether that's with a subscription or API key"
- Provider pattern: Go-native defaults, external command overrides
- Two provider contracts (Type A and Type B) — honest modeling of judgment calls vs. interactive sessions
- Three pipeline phases (Approach / Ascent / Send) — users choose depth per phase
